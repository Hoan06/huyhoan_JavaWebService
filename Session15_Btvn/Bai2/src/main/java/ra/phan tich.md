Sự khác biệt về bảo vệ CSRF: Web truyền thống vs REST APIĐể hiểu tại sao có sự xung đột, chúng ta cần nhìn vào bản chất cách hai kiến trúc này duy trì trạng thái của người dùng:Tiêu chíỨng dụng Web truyền thống (Session-based)REST API (Stateless/Token-based)Cơ chế xác thựcSử dụng Cookie (JSESSIONID) tự động gửi kèm theo mỗi Request từ Trình duyệt.Sử dụng Token (JWT, OAuth2) đính kèm thủ công vào HTTP Header (thường là Authorization: Bearer <token>).Mối đe dọa CSRFNguy cơ cao. Vì trình duyệt tự động gửi Cookie kèm theo request, một trang web độc hại có thể lừa người dùng gửi request giả mạo tới server game/ngân hàng mà họ đã đăng nhập trước đó.Nguy cơ cực thấp (hoặc không có). Mobile App hoặc Single Page App (SPA) không tự động đính kèm Token từ Header nếu không có mã code xử lý cụ thể từ phía Client.Giải pháp bảo vệBắt buộc dùng CSRF Token. Server sinh ra một chuỗi ngẫu nhiên, gắn vào Form hoặc Session, yêu cầu Client gửi ngược lại để đối chiếu.Có thể Vô hiệu hóa (Disable) an toàn nếu Token xác thực được lưu ở bộ nhớ ứng dụng (App Memory, LocalStorage) và gửi qua Header thay vì Cookie.2. Hệ lụy của việc vô hiệu hóa CSRF một cách mù quángNếu bạn phát triển một ứng dụng Web truyền thống (render giao diện bằng JSP, Thymeleaf...) mà lại tắt CSRF (csrf().disable()), bạn đang mở toang cánh cửa cho hacker thực hiện cuộc tấn công Cross-Site Request Forgery.Kịch bản tấn công:Người dùng đăng nhập vào trang ngân hàng bank.com, session được lưu trong Cookie trình duyệt.Người dùng vô tình bấm vào một link lạ dẫn đến malicious-site.com.Trang web độc hại chứa một đoạn code ngầm (hoặc một form ẩn tự động submit) gửi một request POST đến bank.com/api/transfer?amount=1000&to=hacker.Trình duyệt tự động đính kèm Cookie session của bank.com vào request đó. Vì hệ thống đã bị tắt CSRF, Server nhận diện Session hợp lệ và thực hiện lệnh chuyển tiền thành công mà người dùng không hề hay biết.
